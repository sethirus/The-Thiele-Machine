\section{Why Three Layers?}

\subsection{How I Built Trust}

I could write a beautiful mathematical definition of how this machine \textit{should} work, but that alone is not enough. A nice specification can still drift from the executable path. What matters is whether the thing \textit{does what I said it does}.

That's why the system was built three ways. Not because I enjoy multiplying work, but because I needed to know. I needed to see the same answer come out of three different execution paths---one that speaks pure math (Coq), one that speaks Python as a protocol wrapper around extracted OCaml, and one that speaks to actual hardware through Kami-generated Verilog.

If all three paths give me the same answer on the checked projection, I have something solid. If they disagree, one of the pipelines is wrong, and I need to find out which one.

\subsection{The Problem of Trust (The Academic Version)}

A formal specification proves properties but doesn't run on real workloads. An executable implementation runs but might contain bugs or semantic drift. How do you trust that implementation matches specification?

\textbf{Answer}: Derive two executable backends from the same formal specification and verify they produce \textit{identical observable results}. The Python layer delegates to Coq-extracted OCaml; the Verilog is generated from Kami modules written in Coq. Both derive from the same source, but the compilation pipelines are different enough that agreement between them catches extraction bugs, synthesis errors, and serialization mismatches. This makes the thesis rebuildable: every layer can be re-derived from Coq, and any mismatch across pipelines is detectable.

In practice: take a short instruction trace, run it through the Coq-extracted interpreter, the Python VM, and the RTL testbench, compare the gate-appropriate observable projection. If any field diverges, treat it as a semantic bug.

\subsection{The Three Layers}

\begin{enumerate}
    \item \textbf{Coq (Formal)}: Defines ground-truth semantics. Every property is machine-checked. Extraction provides a reference evaluator.
    
    \item \textbf{Python (Reference)}: A protocol wrapper around the Coq-extracted OCaml binary. No independent opcode logic---all execution delegates to \path{build/extracted_vm_runner} via subprocess. Useful for debugging, tracing, and experimentation.

    \item \textbf{Verilog (Hardware)}: Generated from Kami specifications in Coq (\path{coq/kami_hw/ThieleCPUCore.v}) via Bluespec compilation. Synthesizable RTL targeting real FPGAs.
\end{enumerate}
Concretely, the formal layer lives in \texttt{coq/kernel/*.v}, the Python reference VM is implemented under \texttt{thielecpu/} (notably \path{thielecpu/vm.py}), and the RTL is under \texttt{thielecpu/hardware/}. Keeping the directory layout explicit matters because it tells a reader exactly where to validate each part of the story.

\subsection{The Isomorphism Invariant}

For instruction traces $\tau$ within the tested opcode subset:
\[
\pi(S_{\text{Coq}}(\tau)) = \pi(S_{\text{Python}}(\tau)) = \pi(S_{\text{Verilog}}(\tau))
\]
where $\pi$ is a gate-specific observable projection (typically \texttt{vm\_pc} and \texttt{vm\_mu}).

This is not aspirational---it's enforced by automated tests. Any divergence is a critical bug.

\textbf{Honest scope caveat:} The Python layer is not an independent implementation. It delegates all execution to the Coq-extracted OCaml binary via subprocess, so $S_{\text{Python}}(\tau) = S_{\text{Coq}}(\tau)$ by construction. The genuine comparison is between the OCaml extraction pipeline and the Kami/Bluespec synthesis pipeline. The tests compare \textit{state projections} rather than full state identity. The project's own audit artifact at \path{artifacts/final_claim_audit/cross_layer_equivalence_scope.json} specifies \texttt{equivalence\_mode: "observable\_only"} and \texttt{full\_state\_identity\_status: "not claimed"}.

Known divergences exist for several opcodes (MDLACC cost charging, ORACLE\_HALTS hardcoded RTL cost, CHECKPOINT no-op in hardware, CALL/RET init-directive format, PDISCOVER/READ\_PORT text format). The cross-layer bisimulation tests in \path{tests/test_cross_layer_bisimulation.py} document these exceptions explicitly. Of the 40 ISA opcodes, 30 have direct cross-layer mu/register comparison (including the bitwise ALU, LUI, TENSOR, and tightened LASSERT path), 8 have cross-layer tests with documented divergences where each layer is tested independently for its known behavior, and 2 (CERTIFY, WRITE\_PORT) are tested per-layer.

The tests compare \textit{state projections} rather than every internal variable. The projections are suite-specific: the compute gate in \path{tests/test_verilog_cosim.py} compares registers and memory, while the cross-layer bisimulation tests in \path{tests/test_cross_layer_bisimulation.py} compare canonicalized module regions from the partition graph. The extracted runner emits a full JSON snapshot (pc, $\mu$, err, regs, mem, CSRs, graph), but the RTL testbench exposes only the fields required by each gate.

\subsubsection{The Isomorphism Contract (Specification)}

\begin{tcolorbox}[thesisbox,colback=blue!5!white,colframe=blue!75!black,title=3-Layer Comparison Contract]
\textbf{Inputs allowed}:
\begin{itemize}
    \item Instruction traces $\tau$ with explicit $\mu$-deltas per instruction
    \item Initial state: registers all zero, memory all zero, $\mu = 0$, partition graph empty
\end{itemize}

\textbf{Outputs compared}:
\begin{itemize}
    \item \textbf{Compute gate}: registers[0:31], memory[0:65535]
    \item \textbf{Partition gate}: canonicalized module regions (via \texttt{normalize\_region})
    \item \textbf{Full gate}: pc, $\mu$, err, regs, mem, csrs, partition graph
\end{itemize}

\textbf{Canonical serialization rules}:
\begin{itemize}
    \item Regions: sorted, deduplicated lists of indices
    \item Integers: 64-bit words with explicit masking (Coq/Python); 32-bit in RTL
    \item Module IDs: monotonic naturals starting from 0
    \item Hash chains: SHA-256 in hex encoding
\end{itemize}

\textbf{Equivalence definition}: Two states are equivalent under projection $\pi$ iff $\pi(s_1) = \pi(s_2)$ as JSON-serialized dictionaries with identical keys and values.
\end{tcolorbox}

\section{The 3-Layer Comparison Architecture}

One specification, three execution paths, one invariant:
\begin{enumerate}
    \item \textbf{Formal Layer (Coq)}: Ground-truth semantics with machine-checked proofs
    \item \textbf{Reference Layer (Python)}: Protocol wrapper around Coq-extracted OCaml binary, with tracing and debugging
    \item \textbf{Physical Layer (Verilog)}: Kami-generated RTL targeting FPGA/ASIC synthesis
\end{enumerate}

The binding constraint is a cross-layer comparison contract: for supported gates and available backends, the state projections must agree across all three layers. The projections are suite-specific (registers/memory for compute traces; module regions for partition traces), while the extracted runner provides a superset of observables that can be compared when a gate requires it.

\begin{quote}
\textit{Author's Note (Devon): I need to be straight about what ``three layers'' actually means here. The Python VM is not an independent implementation. It serializes state, calls the OCaml binary that was extracted from Coq, and deserializes the result. Zero opcode logic in Python. The Verilog is generated from Kami specifications written in Coq. So all three layers derive from the same Coq source. The genuine test is whether two different compilation pipelines---OCaml extraction and Kami/Bluespec synthesis---produce the same observable outputs. When they agree, that's meaningful. When they disagree (and they do, on about half the ISA), those divergences are documented. I originally called these ``independent implementations'' because that sounded better. But that was wrong. They are three views of one specification. The value is pipeline diversity, not implementation diversity.}
\end{quote}

\section{Layer 1: The Formal Kernel (Coq)}

\subsection{Structure of the Formal Kernel}

The formal kernel is organized around a small set of interlocking definitions:
\begin{itemize}
    \item \textbf{State and partition structure}: the record that defines registers, memory, the partition graph, and the $\mu$-ledger.
    \item \textbf{Step semantics}: the active 46-opcode ISA and the inductive transition rules.
    \item \textbf{Logical certificates}: checkers for proofs and models that allow deterministic verification.
    \item \textbf{Conservation and locality}: theorems that enforce $\mu$-monotonicity and observational no-signaling.
    \item \textbf{Receipts and simulation}: trace formats and cross-layer correspondence lemmas.
\end{itemize}
These bullets correspond directly to files: \texttt{VMState.v} defines the state and partitions, \texttt{VMStep.v} defines the ISA and step relation, \texttt{CertCheck.v} defines certificate checkers, and conservation/locality theorems live in files such as \path{MuLedgerConservation.v} and \path{ObserverDerivation.v}. Receipts and simulation correspondences are defined in \path{ReceiptCore.v} and \path{SimulationProof.v}.

The goal is not to “encode” the implementation, but to define a minimal semantics from which every implementation can be reconstructed.

\subsection{The VMState Record}

The state is defined as a record with twelve components:
\begin{lstlisting}
Record VMState := {
  vm_graph : PartitionGraph;
  vm_csrs : CSRState;
  vm_regs : list nat;
  vm_mem : list nat;
  vm_pc : nat;
  vm_mu : nat;
  vm_mu_tensor : list nat;
  vm_err : bool;
  vm_logic_acc : nat;
  vm_mstatus : nat;
  vm_witness : WitnessCounts;
  vm_certified : bool
}.
\end{lstlisting}

\paragraph{Understanding VMState Record:}

\begin{quote}
\textit{Author's Note (Devon): Look, I know what you're thinking. ``Twelve fields? That's it?'' Yeah. That's it. Every computation this machine does boils down to shuffling values between these twelve buckets. It looks like a lot. But the whole thing is: move bits, track cost. That is all computation ever was. The ledger is just the part that was always there and nobody was writing down.}
\end{quote}

\textbf{This is the complete VM state} --- everything needed to simulate one step.

\textbf{Field-by-Field Breakdown:}
\begin{itemize}
    \item \textbf{vm\_graph : PartitionGraph}: The partition decomposition
    \begin{itemize}
        \item Tracks which modules own which memory/register addresses
        \item Contains axiom sets per module
        \item \textbf{Type}: Defined earlier as \texttt{Record PartitionGraph := \{pg\_next\_id; pg\_modules; pg\_next\_morph\_id; pg\_morphisms\}}
    \end{itemize}
    \item \textbf{vm\_csrs : CSRState}: Control and Status Registers
    \begin{itemize}
        \item Certificate address, privilege level, exception vectors
        \item Analogous to RISC-V CSR file
        \item \textbf{Type}: Another record defined in \texttt{coq/kernel/VMState.v}
    \end{itemize}
    \item \textbf{vm\_regs : list nat}: General-purpose register file
    \begin{itemize}
        \item 32 registers (standard RISC-V count)
        \item Each entry is a natural number (unbounded in Coq)
        \item Hardware masks to 64 bits via \texttt{word64} function (RTL projects to 32 bits)
    \end{itemize}
    \item \textbf{vm\_mem : list nat}: Data memory
    \begin{itemize}
        \item 65,536 words (matching \texttt{MEM\_SIZE} in \path{coq/kernel/VMState.v})
        \item Separate from instruction memory (Harvard architecture)
    \end{itemize}
    \item \textbf{vm\_pc : nat}: Program Counter
    \begin{itemize}
        \item Points to current instruction
        \item Increments by 1 after each step (instructions are unit-indexed in formal model)
        \item Hardware uses byte addressing (increments by 4)
    \end{itemize}
    \item \textbf{vm\_mu : nat}: The $\mu$-ledger accumulator
    \begin{itemize}
        \item Cumulative information cost
        \item Monotonically increasing (never decreases)
        \item \textbf{Core Invariant}: Kernel proofs show this can only grow
    \end{itemize}
    \item \textbf{vm\_mu\_tensor : list nat}: The 4$\times$4 cost tensor
    \begin{itemize}
        \item Flattened 16-entry list representing a 4$\times$4 matrix
        \item Tracks inter-category $\mu$-flow between partition, logic, compute, and transfer operations
        \item Exported as \texttt{getMuTensor0}--\texttt{getMuTensor3} in the Kami-extracted RTL
        \item \textbf{Core Invariant}: Tensor entries are monotonically non-decreasing
    \end{itemize}
    \item \textbf{vm\_err : bool}: Error flag
    \begin{itemize}
        \item \texttt{false} = normal operation
        \item \texttt{true} = undefined behavior detected (e.g., invalid opcode)
        \item Once set, VM halts (no further steps possible)
    \end{itemize}
    \item \textbf{vm\_logic\_acc : nat}: Logic accumulator for certificate-bearing operations
    \item \textbf{vm\_mstatus : nat}: Machine status register (hardware mode bits)
    \item \textbf{vm\_witness : WitnessCounts}: Eight CHSH trial counters (same-outcome and different-outcome buckets for each of the four measurement settings)
    \begin{itemize}
        \item Used by \texttt{CHSH\_TRIAL} to accumulate Bell test statistics
        \item Exported as \texttt{getWcSame00}--\texttt{getWcDiff11} in the Kami RTL
    \end{itemize}
    \item \textbf{vm\_certified : bool}: Certification flag set by \texttt{CERTIFY} instruction
    \begin{itemize}
        \item Once set, indicates the VM has committed to its witness data
        \item Requires positive $\mu$-cost (NoFI enforcement)
    \end{itemize}
\end{itemize}

\textbf{Immutability:} Coq records are immutable. Every instruction creates a \emph{new} VMState rather than mutating the old one. This functional style makes proofs tractable.

Each component has canonical width and representation:
\begin{itemize}
    \item \textbf{vm\_regs}: 32 registers (matching RISC-V convention)
    \item \textbf{vm\_mem}: 65,536 words of data memory
    \item \textbf{vm\_pc}: Program counter (modeled as a natural in proofs; masked to a fixed width in hardware)
    \item \textbf{vm\_mu}: $\mu$-ledger accumulator (modeled as a natural; exported at fixed width in hardware)
    \item \textbf{vm\_err}: Boolean error latch
\end{itemize}
In Coq, the register file and memory are lists, with indices masked by \texttt{reg\_index} and \texttt{mem\_index} in \texttt{coq/kernel/VMState.v}. This makes “out-of-range” indices deterministic and matches the fixed-width semantics of the RTL, where bit widths enforce modular addressing.

\subsection{The Partition Graph}

\begin{lstlisting}
Record PartitionGraph := {
  pg_next_id       : ModuleID;
  pg_modules       : list (ModuleID * ModuleState);
  pg_next_morph_id : MorphismID;
  pg_morphisms     : list (MorphismID * MorphismState)
}.

Record ModuleState := {
  module_region    : list nat;
  module_axioms    : AxiomSet;
  module_mu_tensor : list nat
}.
\end{lstlisting}

\paragraph{Understanding the Partition Graph Data Structures:}
\textbf{PartitionGraph Record:}
\begin{itemize}
    \item \textbf{pg\_next\_id}: Monotonically increasing counter for assigning new ModuleIDs
    \begin{itemize}
        \item Ensures uniqueness: each module gets a distinct ID
        \item Never decreases: proves forward-only allocation
        \item Type: \texttt{ModuleID} (alias for \texttt{nat})
    \end{itemize}
    \item \textbf{pg\_modules}: Association list mapping IDs to module states
    \begin{itemize}
        \item Type: \texttt{list (ModuleID * ModuleState)}
        \item Pairs: \texttt{(id, state)} entries
        \item Lookup: Linear search (O(n)) but simple and verifiable
    \end{itemize}
    \item \textbf{pg\_next\_morph\_id}: Monotonically increasing counter for assigning new MorphismIDs
    \begin{itemize}
        \item Same freshness guarantee as \texttt{pg\_next\_id}, applied to arrows
        \item Type: \texttt{MorphismID} (alias for \texttt{nat})
    \end{itemize}
    \item \textbf{pg\_morphisms}: Association list mapping MorphismIDs to morphism states
    \begin{itemize}
        \item Type: \texttt{list (MorphismID * MorphismState)}
        \item Each entry records source module, target module, cell-to-cell coupling, and certification status
        \item Required for the MORPH, COMPOSE, MORPH\_ID, MORPH\_DELETE, MORPH\_ASSERT, MORPH\_TENSOR, MORPH\_GET opcodes
    \end{itemize}
\end{itemize}

\textbf{ModuleState Record:}
\begin{itemize}
    \item \textbf{module\_region}: List of register/memory addresses owned by this partition
    \begin{itemize}
        \item Example: \texttt{[32, 33, 34]} means module owns registers r32-r34
        \item Disjointness: No two modules can share addresses
        \item Type: \texttt{list nat} (natural numbers = addresses)
    \end{itemize}
    \item \textbf{module\_axioms}: Set of logical constraints for this partition
    \begin{itemize}
        \item Type: \texttt{AxiomSet} (list of SMT-LIB strings)
        \item Example: \texttt{[(assert (>= x 0)), (assert (< x 100))]}
        \item Verified on-chip by the logic engine FSM against program-provided certificates stored in \texttt{vm\_mem}
    \end{itemize}
    \item \textbf{module\_mu\_tensor}: Per-module 4$\times$4 metric tensor
    \begin{itemize}
        \item 16-entry list (row-major flattening of a 4$\times$4 matrix)
        \item Tracks inter-category $\mu$-flow local to this module
        \item Set by \texttt{TENSOR\_SET}, read by \texttt{TENSOR\_GET}
    \end{itemize}
\end{itemize}

\textbf{Physical Interpretation:} The partition graph is the \emph{structural currency}:
\begin{itemize}
    \item \textbf{Modules}: Independent "banks" that own state
    \item \textbf{Regions}: Physical addresses controlled by each module
    \item \textbf{Axioms}: Logical "knowledge" constraining possible values
    \item \textbf{Operations}: Transfer ownership or split/merge banks
\end{itemize}

\textbf{Why This Design?}
\begin{enumerate}
    \item \textbf{Simplicity}: Association lists are easier to prove correct than hash tables
    \item \textbf{Immutability}: Functional updates create new graphs (no mutation)
    \item \textbf{Verifiability}: Linear structure makes proofs tractable
    \item \textbf{Cross-pipeline consistency}: The OCaml extraction and Kami/Bluespec synthesis derive from these same definitions, and observable projections are tested for agreement
\end{enumerate}

Key operations:
\begin{itemize}
    \item \texttt{graph\_pnew}: Create or find module for region
    \item \texttt{graph\_psplit}: Split module by predicate
    \item \texttt{graph\_pmerge}: Merge two disjoint modules
    \item \texttt{graph\_lookup}: Retrieve module by ID
    \item \texttt{graph\_add\_axiom}: Add logical constraint to module
\end{itemize}
In the Python reference VM (\path{thielecpu/vm.py}), these same operations are implemented on a \texttt{RegionGraph} plus a parallel bitmask representation (\texttt{partition\_masks}) to make the RTL mapping explicit. The graph methods enforce the same disjointness and ID discipline as the Coq definitions so that the projection used for cross-layer checks is identical.

\subsection{The Step Relation}

The step relation is an inductive predicate with constructors covering the active opcode set. Several opcodes have both success and failure constructors (e.g., \texttt{step\_psplit} and \texttt{step\_psplit\_failure}), so the constructor count exceeds the opcode count. Each constructor states exact preconditions and the resulting next state:
\begin{lstlisting}
Inductive vm_step : VMState -> vm_instruction -> VMState -> Prop := 
| step_pnew : forall s region cost graph' mid,
    graph_pnew s.(vm_graph) region = (graph', mid) ->
    vm_step s (instr_pnew region cost)
      (advance_state s (instr_pnew region cost) graph' s.(vm_csrs) s.(vm_err))
| step_psplit : forall s m left right cost g' l' r',
    graph_psplit s.(vm_graph) m left right = Some (g', l', r') ->
    vm_step s (instr_psplit m left right cost)
      (advance_state s (instr_psplit m left right cost) g' s.(vm_csrs) s.(vm_err))
...
\end{lstlisting}

\paragraph{Understanding the Step Relation:}
\textbf{Inductive Type Signature:}
\begin{itemize}
    \item \textbf{vm\_step : VMState -> vm\_instruction -> VMState -> Prop}
    \item Takes: current state, instruction, next state
    \item Returns: \texttt{Prop} (logical proposition, not a value)
    \item \textbf{Meaning}: "It is valid to transition from state 1 to state 2 via this instruction"
\end{itemize}

\textbf{Constructor Anatomy (step\_pnew):}
\begin{enumerate}
    \item \textbf{forall s region cost graph' mid}: Universally quantified variables
    \begin{itemize}
        \item \texttt{s}: Current state (input)
        \item \texttt{region, cost}: Instruction parameters
        \item \texttt{graph', mid}: Outputs from graph operation (existential witnesses)
    \end{itemize}
    \item \textbf{Premise}: \texttt{graph\_pnew s.(vm\_graph) region = (graph', mid)}
    \begin{itemize}
        \item The graph operation must succeed
        \item Produces new graph \texttt{graph'} and module ID \texttt{mid}
    \end{itemize}
    \item \textbf{Conclusion}: \texttt{vm\_step s (instr\_pnew ...) (advance\_state ...)}
    \begin{itemize}
        \item Transition from \texttt{s} to updated state
        \item \texttt{advance\_state} helper increments PC and updates $\mu$
    \end{itemize}
\end{enumerate}

\textbf{Constructor Anatomy (step\_psplit):}
\begin{itemize}
    \item \textbf{Option Type}: \texttt{graph\_psplit} returns \texttt{Option} (may fail)
    \item \textbf{Some (g', l', r')}: Pattern match on success case
    \begin{itemize}
        \item \texttt{g'}: New graph after split
        \item \texttt{l', r'}: IDs of left and right modules created
    \end{itemize}
    \item \textbf{Failure Case}: If \texttt{graph\_psplit} returns \texttt{None}, no rule fires (stuck state)
\end{itemize}

\textbf{Why Inductive?} This isn't executable code—it's a \emph{specification}:
\begin{itemize}
    \item \textbf{Relational}: Describes what transitions are valid, not how to compute them
    \item \textbf{Non-determinism}: Multiple rules might apply (though VM is deterministic)
    \item \textbf{Proof Target}: Properties are proven about this relation (safety, progress)
\end{itemize}

\textbf{Step constructors} covering the active opcode set (some have success/failure variants):
\begin{itemize}
    \item Partition ops: PNEW, PSPLIT, PMERGE, PDISCOVER
    \item Logic ops: LASSERT, LJOIN, REVEAL
    \item Transfer \& XOR ops: XFER, XOR\_LOAD, XOR\_ADD, XOR\_SWAP, XOR\_RANK
    \item Scalar ops: LOAD, STORE, ADD, SUB, LOAD\_IMM
    \item Control flow: JUMP, JNEZ, CALL, RET, HALT
    \item Measurement: MDLACC, CHSH\_TRIAL, EMIT, ORACLE\_HALTS
    \item Each constructor specifies exact preconditions (when instruction can execute) and postconditions (resulting state)
\end{itemize}

The \texttt{advance\_state} helper atomically updates PC and $\mu$:
\begin{lstlisting}
Definition advance_state (s : VMState) (instr : vm_instruction)
  (graph' : PartitionGraph) (csrs' : CSRState) (err' : bool) : VMState :=
  {| vm_graph := graph';
     vm_csrs := csrs';
     vm_regs := s.(vm_regs);
     vm_mem := s.(vm_mem);
     vm_pc := S s.(vm_pc);
     vm_mu := apply_cost s instr;
     vm_mu_tensor := s.(vm_mu_tensor);
     vm_err := err';
     vm_logic_acc := s.(vm_logic_acc);
     vm_mstatus := s.(vm_mstatus);
     vm_witness := s.(vm_witness);
     vm_certified := s.(vm_certified) |}.
\end{lstlisting}

\paragraph{Understanding advance\_state:}
\textbf{Purpose:} Centralized state update logic—ensures PC and $\mu$ always advance correctly.

\textbf{Parameters:}
\begin{itemize}
    \item \textbf{s}: Current VMState
    \item \textbf{instr}: Instruction being executed (needed for \texttt{apply\_cost})
    \item \textbf{graph'}: New partition graph (updated by instruction)
    \item \textbf{csrs'}: New CSR state (may be modified by LASSERT, etc.)
    \item \textbf{err'}: New error flag (true if instruction failed)
\end{itemize}

\textbf{Record Construction Line-by-Line:}
\begin{enumerate}
    \item \textbf{vm\_graph := graph'}: Use new partition graph
    \item \textbf{vm\_csrs := csrs'}: Update control/status registers
    \item \textbf{vm\_regs := s.(vm\_regs)}: Preserve registers (unchanged by partition ops)
    \item \textbf{vm\_mem := s.(vm\_mem)}: Preserve memory
    \item \textbf{vm\_pc := S s.(vm\_pc)}: Increment program counter (Coq successor, equivalent to +1)
    \item \textbf{vm\_mu := apply\_cost s instr}: Add instruction's $\mu$-cost to scalar ledger
    \item \textbf{vm\_mu\_tensor := s.(vm\_mu\_tensor)}: Pass through the cost tensor unchanged (the variant \texttt{advance\_state\_reveal} uses \texttt{vm\_mu\_tensor\_add\_at} to update a specific tensor entry)
    \item \textbf{vm\_err := err'}: Set error flag (used for undefined behavior)
    \item \textbf{vm\_logic\_acc := s.(vm\_logic\_acc)}: Preserve logic accumulator
    \item \textbf{vm\_mstatus := s.(vm\_mstatus)}: Preserve machine status
    \item \textbf{vm\_witness := s.(vm\_witness)}: Preserve CHSH witness counters (\texttt{step\_chsh\_trial} updates these separately)
    \item \textbf{vm\_certified := s.(vm\_certified)}: Preserve certification flag (\texttt{step\_certify} sets this separately)
\end{enumerate}

\textbf{Key Function: apply\_cost:}
\begin{itemize}
    \item Extracts the \texttt{mu\_delta} field from \texttt{instr}
    \item Adds it to current $\mu$: \texttt{s.(vm\_mu) + instr.mu\_delta}
    \item \textbf{Monotonicity}: Since \texttt{mu\_delta} is always non-negative, $\mu$ never decreases
\end{itemize}

\textbf{Atomicity:} All updates happen "simultaneously"—no intermediate states:
\begin{itemize}
    \item PC increments exactly when $\mu$ increases
    \item Graph update and $\mu$ charge are inseparable
    \item \textbf{Prevents}: "Free" operations where PC advances without $\mu$ cost
\end{itemize}

\textbf{Register/Memory Variant:} The function \texttt{advance\_state\_rm} (mentioned next) additionally updates \texttt{vm\_regs} and \texttt{vm\_mem} for data-moving instructions like \texttt{XOR\_LOAD} and \texttt{XFER}.
The existence of \texttt{advance\_state\_rm} in \texttt{coq/kernel/VMStep.v} is equally important: register- and memory-modifying instructions (such as \texttt{XOR\_LOAD} and \texttt{XFER}) use a variant that updates \texttt{vm\_regs} and \texttt{vm\_mem} explicitly, so these updates are part of the inductive semantics rather than encoded as side effects.

\textbf{Why Extract?}
\begin{itemize}
    \item \textbf{Proof $\rightarrow$ Program}: Logic verified in Coq becomes runnable code
    \item \textbf{Reference Implementation}: Extracted code is the "ground truth" semantics
    \item \textbf{Testing Oracle}: Python and Verilog implementations are checked against it
    \item \textbf{No Trust Gap}: OCaml code inherits correctness from Coq proofs (modulo extraction bugs)
\end{itemize}

\textbf{Performance vs. Correctness:}
\begin{itemize}
    \item \textbf{Correct}: Extracted code is \emph{provably correct}---it matches the formal model exactly
    \item \textbf{Peano trap}: Coq's \texttt{nat} extracts to unary arithmetic by default. Without the \texttt{Extract Constant Nat.*} and \texttt{word64\_*} overrides in \path{coq/Extraction.v}, basic arithmetic would trigger deeply recursive calls and crash with a stack overflow
    \item \textbf{Use Case}: Validation oracle, not production runtime
\end{itemize}

\textbf{The Extraction Pipeline:}
Extraction is not a single command---it's a reproducible pipeline managed by \path{scripts/forge\_artifact.sh}:
\begin{enumerate}
    \item \textbf{Compile Coq}: \texttt{make -C coq Extraction.vo}
    \item \textbf{Native overrides}: The \texttt{Extract Constant} directives in \path{coq/Extraction.v} replace Peano-recursive \texttt{Nat.*} functions with native \texttt{int} operations (\texttt{add = (+)}, \texttt{mul = ( * )}, \texttt{sub}, \texttt{eqb}) and route \texttt{word64\_*} operations through OCaml's \texttt{Int64} module for 63-bit operational fidelity
    \item \textbf{Compile runner}: \texttt{ocamlfind ocamlc -package str -linkpkg build/thiele\_core.ml build/extracted\_vm\_runner.ml -o build/extracted\_vm\_runner}
    \item \textbf{Result}: A standalone binary that consumes instruction traces and emits JSON state snapshots
\end{enumerate}

\textbf{The Three-Way Check:}
\[
\text{Coq Semantics} \xrightarrow{\text{extract}} \text{OCaml} \longleftrightarrow \text{Python} \longleftrightarrow \text{Verilog}
\]
Extracted OCaml serves as the bridge connecting formal proofs to executable implementations.

The extracted code compiles to a small runner, which serves as an oracle for Python/Verilog comparison.
The runner consumes traces and emits a JSON snapshot of the observable fields. This makes it possible to compare the extracted semantics to the Python VM and RTL without invoking Coq at runtime; the extraction step freezes the semantics into a standalone artifact.

\subsection{Architecture Overview}

The reference VM is optimized for correctness and observability rather than performance. Its purpose is to be readable and to expose every state transition for inspection and replay.

\textbf{Important architectural note:} The Python VM (\path{thielecpu/vm.py}) does not re-implement opcode semantics. All VM execution is delegated to the Coq-extracted OCaml runner binary (\path{build/extracted\_vm\_runner}) via subprocess. The Python layer handles state serialization, deserialization, and tooling (assembler, debugger, receipts) but contains no independent opcode logic. This ensures that the Python path is semantically identical to the Coq extraction by construction, not by parallel implementation.

\subsubsection{Core Components}

The reference VM is structured around:
\begin{itemize}
    \item \textbf{State}: a dataclass mirroring the formal record (registers, memory, CSRs, partition graph, $\mu$-ledger).
    \item \textbf{ISA decoding}: a compact representation of the active opcode set.
    \item \textbf{Partition operations}: creation, split, merge, and discovery.
    \item \textbf{Receipt tooling}: TRS-1.0 artifact and execution receipts, plus historical step-receipt design notes.
\end{itemize}

\subsubsection{The VM Class}

\begin{lstlisting}
class VM:
    """Thin compatibility wrapper around vm_run.

    All execution is delegated to build/extracted_vm_runner
    via vm_run(). No hand-written opcode bodies, no
    independent semantics.
    """
    def __init__(self, state: Optional[VMState] = None):
        self.state: VMState = (
            state if state is not None else VMState.default()
        )

    def apply(self, instr: Dict[str, Any]) -> None:
        self.state = vm_apply(self.state, instr)

    def run_program(self, program: List[Dict[str, Any]],
                    max_steps: int = 10000) -> VMState:
        self.state = vm_run(self.state, program, max_steps=max_steps)
        return self.state
\end{lstlisting}

\paragraph{Understanding the Python VM Class:}
\textbf{One Field:}
\begin{itemize}
    \item \textbf{state: VMState}: The complete VM state, mirroring the Coq \texttt{VMState} record field-for-field (twelve fields: \texttt{vm\_pc}, \texttt{vm\_mu}, \texttt{vm\_err}, \texttt{vm\_regs}, \texttt{vm\_mem}, \texttt{vm\_csrs}, \texttt{vm\_graph}, \texttt{vm\_mu\_tensor}, \texttt{vm\_logic\_acc}, \texttt{vm\_mstatus}, \texttt{vm\_witness}, \texttt{vm\_certified}).
    \item This is the \emph{only} field. No register file, no separate memory array, no sandbox globals. Everything lives inside VMState.
\end{itemize}

\textbf{Methods:}
\begin{itemize}
    \item \textbf{apply(instr)}: Serialize one instruction, call the OCaml binary, parse the JSON output back into a VMState. The Python code never interprets the opcode itself.
    \item \textbf{run\_program(program, max\_steps)}: Serialize a full instruction trace, call the OCaml binary once, get back the final state.
\end{itemize}

\textbf{Why This Design?}
\begin{itemize}
    \item \textbf{No semantic drift}: If the Python code contained its own opcode logic, it would need to be kept in sync with Coq. By delegating to the extracted binary, the Python path is semantically identical to the Coq extraction by construction.
    \item \textbf{Single source of truth}: All opcode semantics live in \texttt{coq/kernel/VMStep.v}. The OCaml binary is a direct extraction of those definitions. The Python layer is pure plumbing.
    \item \textbf{Tested}: The file header says \texttt{DO NOT EDIT. GENERATED FROM COQ PROOFS by scripts/forge\_vm.py}---the entire module is machine-generated.
\end{itemize}

\subsection{State Representation}

The reference state mirrors the formal definition directly. The VMState dataclass is auto-generated by \texttt{scripts/forge\_vm.py} to match the Coq record field-for-field:
\begin{lstlisting}
@dataclass
class VMState:
    vm_pc: int = 0
    vm_mu: int = 0
    vm_err: bool = False
    vm_regs: List[int] = field(default_factory=lambda: [0] * REG_COUNT)
    vm_mem: List[int] = field(default_factory=lambda: [0] * MEM_SIZE)
    vm_csrs: CSRState = field(default_factory=CSRState)
    vm_graph: partitionGraph = field(default_factory=partitionGraph)
    vm_mu_tensor: List[int] = field(default_factory=lambda: [0] * TENSOR_SIZE)
    vm_logic_acc: int = 0
    vm_mstatus: int = 0
    vm_witness: List[int] = field(default_factory=lambda: [0] * 8)
    vm_certified: bool = False
\end{lstlisting}

\paragraph{Understanding the VMState Dataclass:}
\textbf{Twelve fields, one-to-one with Coq:}
\begin{itemize}
    \item \textbf{vm\_pc, vm\_mu, vm\_err}: Program counter, $\mu$-ledger accumulator, error flag
    \item \textbf{vm\_regs}: 32 registers (\texttt{REG\_COUNT = 32})
    \item \textbf{vm\_mem}: 65,536 words of data memory (\texttt{MEM\_SIZE = 65536})
    \item \textbf{vm\_csrs}: Control and status registers (\texttt{CSRState} sub-dataclass)
    \item \textbf{vm\_graph}: Partition graph (\texttt{partitionGraph} sub-dataclass, containing \texttt{pg\_next\_id} and \texttt{pg\_modules})
    \item \textbf{vm\_mu\_tensor}: Flattened 4$\times$4 cost tensor (\texttt{TENSOR\_SIZE = 16})
    \item \textbf{vm\_logic\_acc, vm\_mstatus}: Logic accumulator and machine status
    \item \textbf{vm\_witness}: Eight CHSH trial counters (same/different outcome buckets)
    \item \textbf{vm\_certified}: Certification flag set by \texttt{CERTIFY}
\end{itemize}

\textbf{Constants (top of file):}
\begin{lstlisting}
REG_COUNT: int = 32
MEM_SIZE: int = 65536
TENSOR_SIZE: int = 16
WORD64_MASK: int = 0xFFFFFFFFFFFFFFFF
\end{lstlisting}

\textbf{Serialization:} The \texttt{to\_json()} method packs all twelve fields into a compact JSON string with a keyed FNV-1a integrity MAC, and \texttt{from\_json()} deserializes it back. This is the wire format between Python and the OCaml binary---the runner parses this JSON, executes instructions, and emits an updated JSON snapshot.

\textbf{Isomorphism Mapping:}
\[
\begin{array}{rcl}
\texttt{Coq VMState} & \longleftrightarrow & \texttt{Python VMState} \\
\texttt{vm\_graph} & \longleftrightarrow & \texttt{vm\_graph (partitionGraph)} \\
\texttt{vm\_mu} & \longleftrightarrow & \texttt{vm\_mu} \\
\texttt{vm\_csrs} & \longleftrightarrow & \texttt{vm\_csrs (CSRState)} \\
\texttt{vm\_witness} & \longleftrightarrow & \texttt{vm\_witness} \\
\end{array}
\]
The field names are identical across layers. No translation table is needed---what you see in Coq is what you get in Python.

\subsection{The $\mu$-Ledger}

In the current architecture, $\mu$-accounting is handled by a single integer field (\texttt{vm\_mu}) that mirrors the Coq \texttt{vm\_mu : nat}. There is no separate ledger object---the OCaml extracted runner computes $\mu$-updates via \texttt{apply\_cost} (defined in \texttt{VMStep.v}), and the Python layer simply reads the result from the JSON output.

The Python VM also carries \texttt{vm\_mu\_tensor}, a 16-entry list representing the 4$\times$4 inter-category cost tensor. Together, the scalar \texttt{vm\_mu} and the tensor \texttt{vm\_mu\_tensor} provide a complete picture of information cost: the scalar tracks the global cumulative, and the tensor tracks how cost distributes across partition, logic, compute, and transfer categories.

\textbf{Key constant:}
\begin{lstlisting}
WORD64_MASK: int = 0xFFFFFFFFFFFFFFFF
\end{lstlisting}

The 64-bit mask matches the Coq kernel's \texttt{word64} function. The Verilog RTL operates on 32-bit words (\texttt{WordSz := 32} in \texttt{ThieleTypes.v}), so cross-layer comparison masks to the intersection width.

\textbf{Why a single integer instead of a ledger class?}
\begin{enumerate}
    \item \textbf{Semantic identity}: The Coq definition uses \texttt{vm\_mu : nat}. One integer, not a disaggregated pair. Matching it exactly prevents translation bugs.
    \item \textbf{Monotonicity}: \texttt{apply\_cost} in \texttt{VMStep.v} adds the instruction's cost to the current $\mu$. The cost is always non-negative, so $\mu$ never decreases.
    \item \textbf{Delegation}: Since all opcode execution runs in the OCaml binary, the Python layer never computes a $\mu$-delta itself---it just reads the post-state.
\end{enumerate}

\subsection{Partition Operations}

\subsubsection{Instruction Serialization}

Since all opcode semantics live in the OCaml binary, the Python VM's primary job for partition operations is \emph{serialization}: converting a Python instruction dictionary into the text format the runner expects. Each opcode has a serialization arm:

    ...  # 46 opcode arms total

\paragraph{Understanding the Serialization Architecture:}
\textbf{Why serialize instead of execute?}
\begin{itemize}
    \item The OCaml binary (\path{build/extracted\_vm\_runner}) is the authoritative implementation. It was extracted directly from Coq proofs. If Python re-implemented the partition graph operations, any divergence would be an isomorphism bug.
    \item The serialization format is simple: \texttt{OPCODE arg1 arg2 ... cost}. Lists use curly braces: \texttt{\{0,1,2\}}.
    \item A complete instruction trace is written to a temporary file, the runner is invoked via \texttt{subprocess.run()}, and the JSON output is parsed back into a \texttt{VMState}.
\end{itemize}

\subsubsection{Module Creation (PNEW)}

In the Coq kernel, \texttt{graph\_pnew} creates or finds a module for a region. The Python VM exercises this by serializing a \texttt{PNEW} instruction:
\begin{lstlisting}
vm = VM()
vm.apply({"op": "pnew", "region": [0, 1, 2], "cost": 1})
# Under the hood: sends "PNEW {0,1,2} 1" to OCaml runner
# Runner calls graph_pnew, returns updated VMState as JSON
\end{lstlisting}

\paragraph{Understanding PNEW Semantics (from Coq):}
\begin{itemize}
    \item \textbf{Idempotent}: If a module already owns the exact region, return the existing ID. No duplicate creation, no double charging.
    \item \textbf{$\mu$-cost}: Charged by \texttt{apply\_cost} in \texttt{VMStep.v}. The cost field in the instruction determines how much $\mu$ increases.
    \item \textbf{Result}: The \texttt{vm\_graph} field of the returned VMState contains the new module with its assigned ID.
\end{itemize}

\subsection{Port and Heap Operations}

The ISA includes five I/O and heap instructions: \texttt{CHECKPOINT} captures a state snapshot for debugging and trace replay; \texttt{READ\_PORT} and \texttt{WRITE\_PORT} provide a channel for external communication (e.g., host callbacks or co-processor integration); and \texttt{HEAP\_LOAD} and \texttt{HEAP\_STORE} extend the flat memory model with a separate heap region. All five are low-cost operations that do not modify the $\mu$-ledger or partition graph unless their operands interact with structure-bearing state.

In the current architecture, these instructions are serialized to the OCaml runner just like all other opcodes:
\begin{lstlisting}
if op == "checkpoint":
    return "CHECKPOINT " + str(instr.get("label", "."))
           + " " + str(instr.get("cost", 0))
if op == "read_port":
    return "READ_PORT " + str(int(instr.get("dst", 0)))
           + " " + str(int(instr.get("channel", 0)))
           + " " + str(int(instr.get("value", 0)))
           + " " + str(int(instr.get("bits", 0)))
           + " " + str(instr.get("cost", 0))
\end{lstlisting}

The runner handles the semantics. The Python layer is responsible for nothing beyond converting the instruction dictionary into the runner's text format.

\subsection{Kami-Generated Architecture}

The hardware is not hand-written Verilog. It is generated through a formally verified pipeline: Coq Kami specifications in \path{coq/kami\_hw/ThieleCPUCore.v} are extracted to OCaml, pretty-printed to Bluespec SystemVerilog, and compiled by the BSC compiler into synthesizable Verilog at \path{thielecpu/hardware/rtl/thiele\_cpu\_kami.v} (7,278 lines).

This means the hardware inherits correctness from the Coq proofs. The key property is that the generated Verilog implements the same step relation as the Coq kernel---not by testing, but by construction.

The generated top-level module is \texttt{mkModule1}:

\paragraph{Understanding the Bluespec Interface:}
\textbf{Action-Method Ports:} Unlike traditional Verilog modules with raw data ports, Bluespec generates an \emph{action-method} interface. Each observable is exposed as a getter method with three signals:
\begin{itemize}
    \item \textbf{EN\_getX}: Enable signal (assert high to read)
    \item \textbf{getX}: The value output (e.g., 32-bit register contents)
    \item \textbf{RDY\_getX}: Ready signal (high when the getter can fire)
\end{itemize}

\textbf{Key Getters:}
\begin{itemize}
    \item \textbf{getPC}: Program counter value
    \item \textbf{getMu}: The $\mu$-accumulator (for cross-pipeline comparison)
    \item \textbf{getErr}: Error code register
    \item \textbf{getHalted}: Halt flag
    \item \textbf{getCertified}: Whether CERTIFY has been executed
    \item \textbf{getWcSame00}--\textbf{getWcDiff11}: Eight CHSH witness counters (same/different outcome buckets for four measurement settings)
\end{itemize}

\textbf{Instruction Loading:}
\begin{itemize}
    \item \textbf{loadInstr\_x\_0 [43:0]}: 44-bit input (12-bit address + 32-bit instruction word)
    \item \textbf{EN\_loadInstr}: Assert to write an instruction into instruction memory
    \item The testbench loads programs by cycling through \texttt{loadInstr} before releasing \texttt{RST\_N}
\end{itemize}

\textbf{Single-Rule Design:} The Kami specification defines one rule (\texttt{step}) that fires on every clock cycle. This compiles to a single Bluespec rule, which generates the Verilog signal \texttt{WILL\_FIRE\_RL\_step}. There is no multi-cycle FSM pipeline---instruction fetch, decode, execute, and writeback all happen combinationally within one clock edge. This is simpler to verify than a pipelined design, though it limits clock frequency.

\textbf{Internal Registers (from Kami):}
\begin{itemize}
    \item \textbf{regs}: 1024-bit flat register file (32 $\times$ 32-bit words)
    \item \textbf{mem}: 4096-entry data memory (BSC RegFile submodule)
    \item \textbf{imem}: 4096-entry instruction memory (BSC RegFile submodule)
    \item \textbf{ptTable}: 2048-bit partition table
    \item \textbf{mt\_arr[0:15]}: Per-module metric tensor array (BSC RegFile)
    \item \textbf{pc, mu, err, halted, certified}: Scalar state registers
    \item \textbf{wc\_same\_00}--\textbf{wc\_diff\_11}: Eight witness counter registers
\end{itemize}

\subsection{Execution Model}

The Kami specification defines one rule: \texttt{step}. This rule reads the instruction at \texttt{imem[pc]}, decodes the opcode from bits [31:24], executes the operation, updates all affected registers, and advances the program counter---all within a single clock cycle.

This is different from a traditional pipelined processor. There are no pipeline stages, no stall logic, no forwarding paths. The trade-off is simplicity (easier to verify, correct ordering by construction) at the cost of clock frequency (the critical path is longer because everything happens combinationally).

The generated Verilog reflects this: you will find a single \texttt{always @(posedge CLK)} block with a large case statement on the opcode field, each branch updating the relevant registers via non-blocking assignments. The signal \texttt{WILL\_FIRE\_RL\_step} is high on every cycle when the core is not halted.

\subsection{Instruction Encoding}

Each 32-bit instruction is decoded into opcode and operands. The fixed-width encoding ensures that hardware and software agree on exact bit-level semantics:
\begin{lstlisting}
wire [7:0] opcode = current_instr[31:24];
wire [7:0] operand_a = current_instr[23:16];
wire [7:0] operand_b = current_instr[15:8];
wire [7:0] operand_cost = current_instr[7:0];
\end{lstlisting}

\paragraph{Understanding Hardware Bitfield Extraction:}
\textbf{What is a \texttt{wire}?} In Verilog, \texttt{wire} represents a combinational connection—pure logic with no memory. Think of it as "always-on" circuitry that instantly reflects its inputs. Contrast with \texttt{reg} (register), which holds state across clock cycles.

\textbf{Bitfield Slicing Syntax:}
\begin{itemize}
    \item \textbf{[7:0]}: Declares an 8-bit wide wire (bits 7 down to 0)
    \item \textbf{current\_instr[31:24]}: Extracts bits 31-24 (inclusive) from the 32-bit instruction
    \item \textbf{Big-Endian Convention}: Most significant bits are numbered highest (bit 31 = leftmost)
\end{itemize}

\textbf{How Extraction Works (Gate-Level):}
\begin{enumerate}
    \item \textbf{No Computation}: This isn't a shift or mask operation at runtime—it's pure wiring
    \item \textbf{Synthesis}: The synthesizer connects wires from \texttt{current\_instr[31]} to \texttt{opcode[7]}, \texttt{current\_instr[30]} to \texttt{opcode[6]}, etc.
    \item \textbf{Zero Latency}: Happens instantly—no clock cycles consumed
    \item \textbf{Zero Area}: No gates needed, just wire routing
\end{enumerate}

\textbf{Field Layout Rationale:}
\begin{itemize}
    \item \textbf{Opcode at Top [31:24]}: Decoded first in the pipeline—putting it in most significant bits allows fast extraction
    \item \textbf{Cost at Bottom [7:0]}: Accessed last (during COMPLETE state)—less timing-critical
    \item \textbf{Fixed 8-bit Fields}: Simplifies decoder logic—no variable-length encoding complexity
\end{itemize}

\textbf{Isomorphism Guarantee:} This same bit layout is defined in:
\begin{itemize}
    \item \textbf{Coq}: Via \texttt{decode\_instruction} function with explicit bit masking
    \item \textbf{Python}: Using struct unpacking or bitwise operations
    \item \textbf{Verilog}: This code
\end{itemize}
All three must produce identical field values given the same 32-bit instruction, so encoding stays consistent across compilation pipelines.

\textbf{Example Decoding:} \texttt{0x01050003}
\begin{itemize}
    \item Opcode = \texttt{0x01} = PNEW
    \item Operand\_a = \texttt{0x05} = register 5
    \item Operand\_b = \texttt{0x00} = (unused for PNEW)
    \item Cost = \texttt{0x03} = 3 $\mu$-bits
\end{itemize}

\subsection{$\mu$-Accumulator Updates}

In the Kami specification (\path{coq/kami\_hw/ThieleCPUCore.v}), every instruction branch adds the cost field to the \texttt{mu} register. The generated Verilog translates this to a non-blocking assignment on every clock edge:
\begin{lstlisting}
// Simplified from generated Verilog (mkModule1):
// On each step, mu is updated:
//   mu <= mu + {24'h0, cost_v};
// where cost_v = current_instr[7:0]
// This matches Coq's apply_cost: s.(vm_mu) + instruction_cost instr
\end{lstlisting}

\paragraph{Understanding the $\mu$-Update:}
\textbf{In the Kami source} (\texttt{ThieleCPUCore.v}), the update is written as:
\begin{verbatim}
Write "mu" <- #mu_v + (ZeroExtend _ #cost_v)
\end{verbatim}
This says: read the current mu register, add the zero-extended cost field, write the result back. The Bluespec compiler translates this into Verilog non-blocking assignment with the appropriate bit widths.

\textbf{Atomicity:} Since the entire step rule fires combinationally, all register updates (PC, mu, regs, err, certified, witness counters) happen simultaneously on the clock edge. There is no intermediate state where PC advanced but mu didn't---matching the Coq semantics exactly.

\textbf{Hardware vs. Coq:}
\begin{itemize}
    \item \textbf{Coq}: \texttt{vm\_mu := apply\_cost s instr} (unbounded natural)
    \item \textbf{Verilog}: 32-bit wrapping addition (overflow at $2^{32}$)
    \item \textbf{Bridge}: \texttt{HardwareBisimulation.v} formalizes the Q16.16 representation and proves correspondence within the 32-bit range
\end{itemize}

\subsection{Cost Accounting in Hardware}

\begin{quote}
\textit{Author's Note (Devon): This is the whole game, right here. Every time you look at something---reveal structure, certify a claim, make a decision that can't be unmade---$\mu$ goes up. One direction. Forever. No refunds. I spent months staring at that property before I understood why it was the right one. Monotonicity is not a feature. It is a consequence of what ``knowing something'' actually means.}
\end{quote}

In the Kami-generated hardware, $\mu$-accounting is handled by a 32-bit register (\texttt{mu}) that is incremented inline within the step rule. There is no separate $\mu$-ALU submodule. The cost field from the instruction word (bits [7:0]) is zero-extended and added to the current mu value on every cycle where an instruction executes.

\textbf{Q16.16 Fixed-Point Concept:} While the Coq kernel formalizes a Q16.16 fixed-point representation in \texttt{VMState.v} (with \texttt{Q16\_SHIFT = 16}, \texttt{Q16\_ONE = 65536}, \texttt{Q16\_MAX = 4294967295}), the actual hardware uses simple integer addition for mu. The Q16.16 formalization appears in \texttt{HardwareBisimulation.v}, which proves correspondence between the Coq model and the RTL's 32-bit arithmetic.

\textbf{Information-Theoretic Operations:} Entropy calculations ($-p \log_2 p$) and mutual information computations are performed in the Coq proofs and the OCaml extracted runner, not in the RTL. The hardware's job is to faithfully execute the ISA and accumulate cost. The proofs connecting mu-cost to information-theoretic quantities live in files like \texttt{MuShannonBridge.v} and \texttt{MuShannonQuantitative.v}.

\textbf{Cross-pipeline comparison:} The RTL's 32-bit mu register corresponds to the lower 32 bits of the Coq model's unbounded \texttt{vm\_mu : nat}. The Python VM's \texttt{vm\_mu} field, masked by \texttt{WORD64\_MASK}, is compared against the RTL output via the testbench. For all practical traces (within the tested opcode subset), the values agree exactly.

\textbf{Hardware mu\_tensor scope:} The RTL contains a 16-entry register array (\texttt{mt\_arr[0:15]}) that serves as a Bianchi conservation ledger---it tracks $\mu$-cost allocated by REVEAL operations and enforces $\sum\text{tensor}[i] \le \mu$ per cycle; violation freezes the CPU. \texttt{TENSOR\_SET} writes the source register value into the tensor entry addressed by the instruction's index field, and \texttt{TENSOR\_GET} reads the tensor entry into the destination register. Both operations are functional in the Kami spec (\texttt{ThieleCPUCore.v}), the generated Verilog, and the cross-layer bisimulation test suite. The hardware tensor is a flat 16-entry array (no per-module dispatch); the Coq kernel manages per-module tensors in the partition graph. The mapping between these two representations is mediated by the abstraction layer in \texttt{Abstraction.v}. The physics proofs (EinsteinEquations4D.v, DiscreteRaychaudhuri.v, CurvedTensorPipeline.v) define curvature computations over Coq reals (\texttt{R}) and abstract module states---they are purely proof-level and are not extracted to executable code. No program has been assembled that performs 4D curvature computation on the Verilog RTL.

\subsection{Logic Engine Interface}

The logic engine is implemented as an on-chip FSM inside \texttt{mkModule1}. The Kami specification in \texttt{ThieleCPUCore.v} defines ten FSM registers that read formula and witness data directly from \texttt{vm\_mem}:
\begin{lstlisting}
// On-chip FSM registers (ThieleCPUCore.v):
//   lassert_phase   -- FSM phase (idle/fetch_formula/fetch_cert/check/done)
//   lassert_kind    -- SAT or UNSAT certificate kind
//   lassert_fbase   -- formula base address (from freg)
//   lassert_cbase   -- certificate base address (from creg)
//   lassert_flen    -- encoded formula-unit count
//   lassert_clen    -- certificate unit count
//   lassert_fptr    -- formula fetch pointer (incremented each cycle)
//   lassert_cptr    -- certificate fetch pointer
//   lassert_fbuf    -- formula unit accumulator
//   lassert_cbuf    -- certificate unit accumulator
\end{lstlisting}

\paragraph{On-Chip Evaluation Protocol:}
\textbf{Operation:} When an \texttt{LASSERT} instruction executes, the FSM reads the formula from \texttt{vm\_mem} at address \texttt{R[freg]}, then reads the witness block from \texttt{R[creg]}. In SAT mode that block contains two assignments back-to-back: a satisfying model and a falsifying countermodel. It verifies both entirely on-chip. The $\mu$-cost charged is $\mathtt{flen} \times 8 + S(\mathtt{cost})$, so description length is reflected in cost while the second witness shows the spend bought a real narrowing.

\textbf{Why Registers Instead of a Submodule?}
\begin{itemize}
    \item \textbf{Kami's model}: Kami generates flat modules with registers and rules. The FSM state is flat registers following Bluespec's composition model.
    \item \textbf{Self-containment}: No external solver is needed---the VM verifies certificates autonomously at runtime from program-supplied data in \texttt{vm\_mem}.
    \item \textbf{Formal verification}: \path{coq/kami\_hw/LogicEngineEquivalence.v} proves that the on-chip FSM evaluation refines the Coq step relation for \texttt{LASSERT}, \texttt{LJOIN}, and related opcodes.
\end{itemize}

\subsection{The Comparison Gate}

The cross-pipeline comparison is verified by a test that:
\begin{enumerate}
    \item Generate instruction trace $\tau$ (from the subset of opcodes with cross-layer agreement)
    \item Execute $\tau$ on Python VM $\rightarrow$ state $S_{\text{py}}$ (delegates to extracted OCaml runner, so $S_{\text{py}} = S_{\text{coq}}$ by construction)
    \item Execute $\tau$ on extracted runner directly $\rightarrow$ state $S_{\text{coq}}$
    \item Execute $\tau$ on Verilog sim $\rightarrow$ state $S_{\text{rtl}}$
    \item Assert $\pi(S_{\text{coq}}) = \pi(S_{\text{rtl}})$ under the gate-specific projection $\pi$
\end{enumerate}
The genuine test is step 5: comparing the OCaml extraction pipeline against the Kami/Bluespec synthesis pipeline. Steps 2-3 are tautological (same binary), but are included for completeness and to catch serialization bugs in the Python wrapper.

\subsection{State Projection}

For comparison, states are projected to canonical summaries tailored to the gate being exercised. The extracted runner emits a full JSON snapshot (pc, $\mu$, err, regs, mem, CSRs, graph), which can be projected down to subsets. The compute gate uses only registers and memory, while the partition gate uses canonicalized module regions. A full projection helper is therefore a \emph{superset} view, not the only comparison performed:
\begin{lstlisting}
def project_state_full(state):
    return {
        "pc": state.pc,
        "mu": state.mu,
        "err": state.err,
        "regs": list(state.regs[:32]),
        "mem": list(state.mem[:65536]),
        "csrs": state.csrs.to_dict(),
        "graph": state.graph.to_canonical(),
    }
\end{lstlisting}

\paragraph{Understanding State Projection:}
\textbf{Purpose:} Converts internal VM state to JSON-serializable dictionary for cross-layer comparison.

\textbf{Dictionary Fields:}
\begin{itemize}
    \item \textbf{"pc": state.pc}: Program counter value (integer)
    \item \textbf{"mu": state.mu}: $\mu$-ledger total (integer or float)
    \item \textbf{"err": state.err}: Error flag (boolean)
    \item \textbf{"regs": list(state.regs[:32])}: First 32 registers as list
    \begin{itemize}
        \item Slice \texttt{[:32]} ensures fixed size
        \item \texttt{list(...)} converts from internal representation
    \end{itemize}
    \item \textbf{"mem": list(state.mem[:65536])}: Full 65,536-word memory
    \begin{itemize}
        \item Full data memory snapshot
    \end{itemize}
    \item \textbf{"csrs": state.csrs.to\_dict()}: CSR snapshot
    \begin{itemize}
        \item Converts CSRState object to dictionary
        \item Includes certificate address, exception vectors, etc.
    \end{itemize}
    \item \textbf{"graph": state.graph.to\_canonical()}: Canonical partition encoding
    \begin{itemize}
        \item Sorts modules by ID
        \item Sorts region addresses within each module
        \item Ensures comparison doesn't fail due to ordering differences
    \end{itemize}
\end{itemize}

\textbf{Canonicalization:} The \texttt{to\_canonical()} call is critical:
\begin{itemize}
    \item Python sets are unordered, Coq lists are ordered
    \item Without canonicalization: $\{1, 2, 3\} \neq \{3, 2, 1\}$ (as JSON)
    \item With canonicalization: Both become \texttt{[1, 2, 3]}
\end{itemize}

\textbf{Projection Strategy:}
\begin{enumerate}
    \item \textbf{Full Projection}: This function — includes all fields
    \item \textbf{Compute Projection}: Only \texttt{\{"regs", "mem"\}} — for ALU tests
    \item \textbf{Partition Projection}: Only \texttt{\{"graph", "mu"\}} — for PNEW/PSPLIT tests
    \item \textbf{Why Multiple?} Different tests care about different state components
\end{enumerate}

\textbf{Comparison Use:} After running same instruction trace on Coq/OCaml runner and Verilog sim:
\begin{verbatim}
coq_state_json = ocaml_runner_output()
rtl_state_json = verilog_sim_output()
assert project(coq_state_json) == project(rtl_state_json)
\end{verbatim}
If any projected field differs, the comparison gate fails. (Note: the Python-vs-Coq comparison is tautological since the Python VM delegates to the same OCaml binary.)

\subsection{The Inquisitor}

\begin{quote}
\textit{Author's Note (Devon): The Inquisitor is my paranoia made into automation. Every time changes get pushed, it checks for admits that might have snuck in, proofs that don't compile, theorems that secretly prove nothing. The Inquisitor (\path{scripts/inquisitor.py}, $\sim$7,400 lines) is a heuristic static analysis tool that scans the Coq source tree for ``proof smells''---patterns that indicate incomplete, vacuous, or dishonest proofs. It goes far beyond simple pattern matching.

\textbf{What it detects} (20+ rule types across three severity levels):

\textbf{HIGH (FORBIDDEN)} --- any of these fails CI:
\begin{itemize}
    \item \texttt{ADMITTED} / \texttt{ADMIT\_TACTIC} / \texttt{GIVE\_UP\_TACTIC}: Incomplete proofs. Zero tolerance.
    \item \texttt{AXIOM\_OR\_PARAMETER}: Undocumented \texttt{Axiom} or \texttt{Parameter} declarations. Every assumption must be explicitly tracked.
    \item \texttt{HYPOTHESIS\_ASSUME}: \texttt{Hypothesis} declarations (functionally equivalent to Axiom).
    \item \texttt{CONTEXT\_ASSUMPTION}: Undocumented \texttt{Context} with \texttt{forall}/arrow types---these are section-local axioms that must be instantiated or documented.
    \item \texttt{PROP\_TAUTOLOGY}: Theorems that are literally \texttt{True}. A theorem that proves nothing is worse than no theorem.
    \item \texttt{IMPLIES\_TRUE\_STMT} / \texttt{LET\_IN\_TRUE\_STMT} / \texttt{EXISTS\_TRUE\_STMT}: Hidden vacuity---statements that \textit{look} like they prove something but end in \texttt{True}.
    \item \texttt{CONST\_Q\_FUN} / \texttt{EXISTS\_CONST\_Q}: Constant probability witnesses (\texttt{fun \_ => 0\%Q}). These are placeholders that trivialize any bound.
    \item \texttt{COMPILATION\_ERROR}: Files that fail to compile.
    \item \texttt{ASSUMPTION\_AUDIT}: Unexpected axioms detected via \texttt{Print Assumptions} in \texttt{coqtop}---a runtime check that the actual axiom dependencies match expectations.
\end{itemize}

\textbf{MEDIUM} --- flagged for review, escalated to HIGH in critical kernel files:
\begin{itemize}
    \item \texttt{COST\_IS\_LENGTH}: Cost defined as list length (often a placeholder).
    \item \texttt{ZERO\_CONST} / \texttt{EMPTY\_LIST}: Trivial constant definitions that may indicate unfinished work.
    \item \texttt{CLAMP\_OR\_TRUNCATION}: Uses of \texttt{Z.to\_nat}, \texttt{Nat.min}, \texttt{Nat.max} without nonnegativity guards---these can silently clamp values and break algebraic laws.
    \item \texttt{COMMENT\_SMELL}: \texttt{TODO}/\texttt{FIXME}/\texttt{WIP} markers in comments.
    \item \texttt{SUSPICIOUS\_SHORT\_PROOF}: Complex-sounding theorems (\texttt{*bound*}, \texttt{*uniqueness*}, \texttt{*conservation*}) with very short proofs.
    \item \texttt{MU\_COST\_ZERO}: $\mu$-cost definitions equal to zero by definition.
    \item \texttt{PHYSICS\_ANALOGY\_CONTRACT}: Physics-named theorems lacking an accompanying invariance lemma.
    \item \texttt{PROBLEMATIC\_IMPORT}: Imports of \texttt{Classical}, \texttt{Decidable}, or \texttt{ProofIrrelevance} that may introduce classical axioms.
\end{itemize}

\textbf{LOW} --- informational:
\begin{itemize}
    \item \texttt{TRIVIAL\_EQUALITY}: Theorems of the form $X = X$ proved by reflexivity.
    \item \texttt{CIRCULAR\_INTROS\_ASSUMPTION}: Tautology-shaped statements proved via \texttt{intros; assumption}.
    \item \texttt{AXIOM\_DOCUMENTED}: Assumptions marked with \texttt{INQUISITOR NOTE} comments---tracked but accepted.
\end{itemize}

\textbf{Critical file escalation.} The Inquisitor maintains two protection tiers:
\begin{itemize}
    \item \textbf{Protected files}: \texttt{CoreSemantics.v}, \texttt{BridgeDefinitions.v}---MEDIUM findings escalate to HIGH.
    \item \textbf{Critical kernel files}: \texttt{VMState.v}, \texttt{VMStep.v}, \texttt{NoFreeInsight.v}, \texttt{MuCostModel.v}, \texttt{TsirelsonUpperBound.v}, and 5 others---extra scrutiny applied.
\end{itemize}

\textbf{Documentation mechanisms.} Two comment conventions let developers explicitly whitelist findings:
\begin{itemize}
    \item \texttt{INQUISITOR NOTE}: Documents why an assumption is accepted (downgrades to LOW).
    \item \texttt{(* SAFE: reason *)}: Whitelists a specific finding within 3 lines (e.g., a \texttt{Z.to\_nat} with an external guard).
\end{itemize}

\textbf{Runtime verification.} Beyond static scanning, the Inquisitor can invoke \texttt{coqtop} to run \texttt{Print Assumptions} on key theorems, verifying that the \textit{actual} axiom dependencies (not just the declared ones) match the documented set. It also verifies a paper symbol map---checking that every theorem cited in this thesis corresponds to a real Coq symbol that compiles.

\textbf{Allowlist.} A JSON allowlist (\path{scripts/inquisitor\_allowlist.json}) documents mathematically-justified exceptions with written justifications for why each finding is acceptable.

The CI gate enforces a single rule: \textbf{zero HIGH findings}. If a single HIGH finding exists anywhere in the Coq tree, CI fails. No exceptions, no overrides.

\end{lstlisting}

\paragraph{Understanding Yosys Synthesis:}
\textbf{Yosys:} Open-source RTL synthesis tool that converts Verilog to gate-level netlists.

\textbf{Command Breakdown:}
\begin{itemize}
    \item \textbf{yosys}: The synthesizer executable
    \item \textbf{-p "..."}: Pass string (execute commands)
    \item \textbf{read\_verilog thiele\_cpu\_kami.v}: Load Kami-generated Verilog source
    \begin{itemize}
        \item Parses file, builds abstract syntax tree
        \item Checks basic syntax errors
    \end{itemize}
    \item \textbf{synth\_ecp5}: Run Lattice ECP5 synthesis flow
    \begin{itemize}
        \item Optimizes for Lattice ECP5 primitives
        \item Maps to LUTs, FFs, BRAM, DSP blocks
    \end{itemize}
    \item \textbf{-top mkModule1}: Specify top-level module name
    \begin{itemize}
        \item Entry point for synthesis (Kami-generated module)
        \item All internal logic is instantiated within this
    \end{itemize}
\end{itemize}

\textbf{Synthesis Steps (Internal):}
\begin{enumerate}
    \item \textbf{Elaboration}: Flatten hierarchy, expand parameters
    \item \textbf{Optimization}: Remove dead code, constant propagation
    \item \textbf{Technology Mapping}: Convert to FPGA primitives
    \begin{itemize}
        \item \texttt{always @(posedge clk)} $\rightarrow$ FDRE (D flip-flop)
        \item \texttt{case} statements $\rightarrow$ LUT6 (6-input LUT)
        \item \texttt{+} operator $\rightarrow$ CARRY4 (fast carry chain)
    \end{itemize}
    \item \textbf{Output}: JSON netlist or EDIF for place-and-route
\end{enumerate}

\textbf{Output Reports:}
\begin{itemize}
    \item \textbf{Resource Usage}: Number of LUTs, FFs, BRAMs
    \item \textbf{Critical Path}: Longest combinational delay
    \item \textbf{Warnings}: Latches inferred, unconnected signals
\end{itemize}

\textbf{Next Steps After Synthesis:}
\begin{enumerate}
    \item \textbf{Place \& Route}: Vivado/ISE assigns physical locations
    \item \textbf{Bitstream Generation}: Creates FPGA configuration file
    \item \textbf{Programming}: Load bitstream onto FPGA via JTAG
\end{enumerate}

\textbf{Alternative Targets:}
\begin{itemize}
    \item \textbf{synth\_ice40}: For Lattice iCE40 FPGAs (smaller, cheaper)
    \item \textbf{synth\_ecp5}: For Lattice ECP5
    \item \textbf{synth\_intel}: For Intel/Altera devices
    \item \textbf{synth}: Generic synthesis (not vendor-specific)
\end{itemize}

\section{Summary}

The 3-layer implementation delivers:
\begin{itemize}
    \item \textbf{Logical Certainty}: Coq proofs guarantee properties hold for all inputs
    \item \textbf{Operational Visibility}: Python traces expose every state transition
    \item \textbf{Physical Realizability}: Verilog synthesizes to real hardware
\end{itemize}

The binding across layers is not aspirational---it's enforced through automated comparison gates on a tested opcode subset. The Inquisitor ensures no admits, documented axioms only, and no semantic divergences ever hit the main branch. The genuine verification value is the agreement between two different compilation pipelines from the same Coq source: OCaml extraction and Kami/Bluespec synthesis. Where these pipelines agree, we have high confidence in the semantics. Where they diverge (documented in the cosim tests), the divergences are explicit and bounded.
